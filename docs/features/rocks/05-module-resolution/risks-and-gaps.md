# ROCKS-05: Risks & Gaps

## Critical Risks

### Risk 1.1: RockspecBridge subprocess latency on large projects
- **Impact**: `RockspecBridge.read()` spawns a Lua subprocess per rockspec. In a workspace with many sub-rocks (10+), the initial scan could take several seconds, during which no rockspec patterns are available.
- **Likelihood**: medium (typical workspaces have 3–10 rocks)
- **Mitigation**: Background computation with atomic swap (design §3.4). First access returns empty patterns; background fills them in. IDE re-indexes when patterns arrive. User sees correct resolution after initial indexing completes (~2–5s on typical project).

### Risk 1.2: Pattern derivation heuristic doesn't match all rockspec layouts
- **Impact**: The algorithm assumes `build.modules` source paths contain the module name as a directory suffix (e.g. `lua/adt/orderedmap.lua` for module `adt.orderedmap`). A rockspec with unconventional mapping (e.g. `["mymod"] = "completely/different/path.lua"`) would not derive a useful pattern.
- **Likelihood**: low (LuaRocks convention and `builtin` build type strongly favor regular directory structures)
- **Mitigation**: Log warning for non-matching entries. Future enhancement: fall back to direct module→file index for entries that don't derive a pattern.

### Risk 1.3: Invalidation race between VFS events and cache
- **Impact**: A rockspec edit triggers `ProjectRootModificationTracker` invalidation. If the cache is being computed (background thread reading other rockspecs), the old computation might overwrite the new invalidation signal.
- **Likelihood**: low (rockspec edits are infrequent; race window is small)
- **Mitigation**: Use `@Volatile` on the cached pattern list + a generation counter. Background computation checks generation before storing results; stale computations are discarded.

## Technical Debt & Future Work
- **TBD: Installed rock patterns** — Derive patterns from rocks in `lua_modules/` so that IDE resolution works even without `setup.lua` in the source path config. Low priority because `setup.lua` is generated by ROCKS-01.
- **TBD: Direct module→file index** — For rockspecs with unconventional mappings (Risk 1.2), a `FileBasedIndex` keyed by module name could provide exact resolution. Deferred because the pattern approach handles >99% of real rockspecs.
- **TBD: Rockspec change notifications** — Show a notification when rockspec patterns change ("New module paths detected. Re-indexing..."). Low priority cosmetic enhancement.
