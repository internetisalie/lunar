# LuaRocks Project Setup Investigation

Based on the article "Lua project setup with LuaRocks" from martin-fieber.de, here are the key findings and recommendations for setting up a Lua project with LuaRocks.

## Overview

This investigation covers the essential steps for setting up a Lua project using LuaRocks for dependency management, including project initialization, dependency handling, module resolution, code quality tools, and editor integration.

## Potential Use Cases

Users developing Lua projects with LuaRocks may encounter various scenarios where specific LuaRocks commands and workflows are applicable:

- **Project Initialization**: Setting up a new Lua project with proper dependency management structure
- **Adding Dependencies**: Installing new libraries and tracking them in the project manifest
- **Removing Dependencies**: Uninstalling packages and updating dependencies list
- **Dependency Resolution**: Ensuring all required packages are installed for development/deployment
- **Version Management**: Specifying and updating Lua version compatibility
- **Local Development**: Configuring module paths for testing and development
- **CI/CD Integration**: Automating dependency installation in build pipelines
- **Publishing Packages**: Creating and distributing LuaRocks packages
- **Environment Configuration**: Setting up editor and tooling integration
- **Code Quality Enforcement**: Integrating formatters and linters into workflow
- **Executable Management**: Installing and using command-line tools from LuaRocks
- **Custom Installation Trees**: Specifying alternative installation directories
- **Development vs Production**: Managing different dependency sets for various environments

## Key Components

### 1. Project Initialization

The `luarocks init` command is the starting point for project setup:

```bash
luarocks init --lua-versions "5.1,5.2,5.3,5.4,5.5"
```

This command:
- Creates a `lua_modules/` directory for local dependency installation
- Sets up a `.luarocks` configuration folder
- Generates project shortcuts for `lua` and `luarocks` commands
- Creates/updates `.gitignore` with appropriate entries
- Generates a `.rockspec` file with base project settings

### 2. Dependency Management

#### Installation
```bash
luarocks install <package_name>
```

#### Tracking Dependencies
Dependencies are tracked in the `.rockspec` file:
```lua
dependencies = {
   "lua >= 5.1, < 5.6",
   "inspect >= 3.1"
}
```

#### Installing Dependencies
```bash
luarocks install --deps-only <project>.rockspec
```

### 3. Module Resolution (Critical Step)

Despite having dependencies installed, Lua needs explicit help to find them:

#### Recommended Solution: Create `src/setup.lua`
```lua
local version = _VERSION:match("%d+%.%d+")

package.path = 'lua_modules/share/lua/' .. version ..
    '/?.lua;lua_modules/share/lua/' .. version ..
    '/?/init.lua;' .. package.path
package.cpath = 'lua_modules/lib/lua/' .. version ..
    '/?.so;' .. package.cpath
```

#### Usage Methods
1. Command line: `lua -l src/setup src/main.lua`
2. Environment variable: `export LUA_INIT="@src/setup.lua"`
3. With direnv: Add to `.envrc` for automatic loading

### 4. Code Quality Tools

#### Auto-formatting (LuaFormatter)
```bash
luarocks install --server=https://luarocks.org/dev luaformatter
lua-format --in-place src/**/*.lua
```
Configuration via `.lua-format` YAML file.

#### Static Analysis (Luacheck)
```bash
luarocks install luacheck
luacheck src
```

### 5. Editor Integration

#### VS Code
- Requires Lua language server + Lua Debug plugins
- May need `require "src/setup"` in main file for module resolution with Lua Debug

#### CLion (JetBrains)
- Uses LSP4IJ plugin + SumnekoLua
- Requires custom build target and run configuration with `LUA_INIT` environment variable

#### Sublime Text 4
- Requires LSP, LSP-lua, and Direnz packages via Package Control
- Project file configuration for build system and LSP settings

#### ZeroBrane
- Works well with `require "src/setup"` in main entry point

### 6. .gitignore Entries
Automatically generated by `luarocks init`:
```
/luarocks
/lua
/lua_modules
/.luarocks
```

### 7. .rockspec File Structure
Complete example:
```lua
rockspec_format = "3.0"
package = "project-name"
version = "dev-1"
source = {
   url = "https://github.com/username/project.git"
}
description = {
   summary = "Project summary",
   homepage = "https://project-homepage.com",
   license = "MIT"
}
dependencies = {
   "lua >= 5.1, < 5.6",
   "inspect >= 3.1"
}
build = {
   type = "builtin",
   modules = {
      main = "src/main.lua"
   }
}
```

### 8. Local Script Execution
After installing executables via LuaRocks:
- Binaries go to `lua_modules/bin/`
- Add to PATH: `export PATH=$PATH:./lua_modules/bin` (especially useful with direnv)

### 9. Rock Trees and Installation Locations
- Default: Packages installed in project's `lua_modules/` when `.luarocks` folder present
- Global: Packages installed system-wide when no `.luarocks` config
- Local (user): `--local` flag installs to user's home directory
- Custom tree: `--tree=lua_modules` specifies installation directory

## Recommendations

1. **Always use `luarocks init`** for proper project scaffolding
2. **Create `src/setup.lua`** for reliable module path resolution
3. **Track all dependencies** in the `.rockspec` file
4. **Validate rockspec files** with `luarocks lint`
5. **Use direnv** for automatic environment variable management (`LUA_INIT` and PATH)
6. **Configure editor integration** to work with the LuaRocks setup
7. **Include code quality tools** (formatter and linter) in development workflow
8. **Consider adding `src/?.lua`** to package.path for inner-project module resolution

## Common Pitfalls

- Forgetting to configure Lua to find locally installed modules (most common issue)
- Not updating the `.rockspec` when adding new dependencies
- Attempting to use globally installed packages when expecting local ones
- Overlooking the need to add LuaRocks binaries to PATH for script execution

This setup provides a solid foundation for Lua development with proper dependency management, reproducible builds, and editor support.
