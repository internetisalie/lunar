# MAINT-01: Kotlin Conversion

## Overview
Six hand-written Java source files remain in an otherwise Kotlin codebase. This feature converts
**four** of them to idiomatic Kotlin per `docs/engineering-contract.md`, with zero behavioral
change. The remaining two тАФ the lexer token-constant interfaces `LuaTokenTypes` and
`LuaCatsTokenTypes` тАФ are **deferred** (see [Out of Scope](#out-of-scope) and **MAINT-19**): the
JFlex-generated lexers `implements` them to inherit their constants, a contract no idiomatic
Kotlin shape preserves without editing the `.flex` sources and regenerating the lexers (which is
the [platform.syntax migration](../requirements.md) itself).

> **Stale-count note:** `docs/status-detail.md` and `CLAUDE.md`'s TODO list both cite a 4-file
> count naming `LuaPsiUtils`, `LuaTokenType`, `LuaCatsElementType`, `LuaPluginDisposable`. A
> `glob '**/*.java'` under `src/main/java/` re-confirms the true on-disk count is **6** (those four
> plus `LuaTokenTypes` and `LuaCatsTokenTypes`). This feature converts the four; MAINT-19 owns the
> other two. The doc-correction task (MAINT-01-10) records this split, not a bare "6".

## Scope

### In Scope
- Convert these **four** hand-written `.java` files under
  `src/main/java/net/internetisalie/lunar/` to Kotlin at the mirrored path under
  `src/main/kotlin/...`, deleting the `.java` originals:
  - `lang/psi/LuaTokenType.java` (MAINT-01-03)
  - `luacats/lang/lexer/LuaCatsElementType.java` (MAINT-01-04)
  - `lang/psi/LuaPsiUtils.java` (MAINT-01-02)
  - `LuaPluginDisposable.java` (MAINT-01-06)
- Preserve all public API surfaces (method/field names, signatures, `toString()` formats) so no
  other compilation unit's source changes are required as a result of the conversion itself.
- Remove dead/commented-out code blocks encountered during conversion (do not port comments
  forward) per the engineering contract's "no unnecessary comments" rule.
- Fix import hygiene (no wildcard imports) and null-safety (no `!!`) in the converted files.

### Out of Scope
- **`lang/lexer/LuaTokenTypes.java` and `luacats/lang/lexer/LuaCatsTokenTypes.java` тАФ DEFERRED to
  MAINT-19, not converted here.** These Java `interface`s hold the lexer token constants, and the
  JFlex-generated lexers inherit those constants by *implementing the interface*:
  - `src/main/kotlin/net/internetisalie/lunar/lang/lexer/lua.flex:17-18` тЖТ
    `%class _LuaLexer` / `%implements FlexLexer, LuaTokenTypes`
  - `src/main/kotlin/net/internetisalie/lunar/luacats/lang/lexer/luacats.flex:24` тЖТ
    `%implements FlexLexer, LuaCatsTokenTypes`
  - generated `src/main/gen/.../lang/lexer/_LuaLexer.java:13` тЖТ
    `class _LuaLexer implements FlexLexer, LuaTokenTypes`, using the constants **bare/unqualified**
    (`_LuaLexer.java:680,685,750,780` тЖТ `return WS; / STRING; / NUMBER; / IDENTIFIER;`)
  - generated `src/main/gen/.../luacats/lang/lexer/_LuaCatsLexer.java:25` тЖТ
    `public class _LuaCatsLexer implements FlexLexer, LuaCatsTokenTypes`

  A Kotlin `object` compiles to a **final class**, which Java cannot `implements`; a Kotlin
  `interface` **cannot hold initialized `val` constants with backing fields**. So neither idiomatic
  Kotlin shape preserves the `implements`-and-inherit-bare-constants contract. Converting these two
  would require editing both `.flex` files (drop `%implements`, qualify every token reference, add
  `@JvmField`/`const`) **and** a manual JFlex regeneration of `_LuaLexer.java` / `_LuaCatsLexer.java`
  per `CLAUDE.md` ("Note on Lexer/Parser Generation") тАФ work that belongs to the platform.syntax
  migration (**MAINT-19**), not this syntax-preserving port.
- The **generated** Java sources under `src/main/gen/` (`LuaElementTypes.java`,
  `LuaCatsElementTypes.java`, `_LuaLexer.java`, `_LuaCatsLexer.java`, parser/PSI impl classes) are
  **not** touched тАФ they are regenerated by the manual IDE/CLI workflow in `CLAUDE.md`.
- Any change to the actual token set, grammar, or PSI tree shape (`lua.bnf` / `lua.flex` /
  `luacats.bnf` / `luacats.flex` are untouched by this feature).
- `LuaIntroduceVariableHandler.kt`'s own `TemplateBuilder.replaceElement(...)` call (line 133) тАФ
  unrelated platform API of the same name, not `LuaPsiUtils.replaceElement`; called out so it is
  not mistaken for a call site during conversion.

## Detailed Specifications

### MAINT-01-02: `LuaPsiUtils.java` тЖТ `LuaPsiUtils.kt`
- **Current**: `src/main/java/net/internetisalie/lunar/lang/psi/LuaPsiUtils.java` тАФ `public final
  class LuaPsiUtils` with these static methods: `nestingLevel(PsiElement): int`,
  `blockNestingLevel(PsiElement): int`, `elementAfter(PsiElement): PsiElement?` (already
  `@Nullable`), `findEnclosingBlock(PsiElement): PsiElement?`, `isValidContainer(PsiElement): boolean`
  (private), `processChildDeclarationsS(...): boolean`, `processChildDeclarations(...): boolean`,
  `getElementLineNumber(PsiElement): int`, `getElementEndLineNumber(PsiElement): int`,
  `createRange(PsiElement): TextRange`, `nodeType(PsiElement): IElementType` (NOT annotated
  `@Nullable` today despite returning `null` at line 208 when `element.getNode()` is null тАФ the one
  signature the conversion must tighten), `findNextSibling(...)`, `findPreviousSibling(...)`,
  `replaceElement(PsiElement, PsiElement): PsiElement`,
  `toPsiElementArray(Collection<? extends PsiElement>): PsiElement[]`,
  `hasDirectChildErrorElements(PsiElement): boolean`.
- **Confirmed dead code, repo-wide**: `grep -rn "LuaPsiUtils\."` across `src/` (excluding the
  file's own definition) returns **zero matches** тАФ none of the 14 public methods has a caller
  anywhere in `src/main`, `src/test`, or `src/integrationTest` today. The conversion preserves the
  full public API 1:1 regardless (do not delete methods as "unused" тАФ that is outside this feature's
  scope; MAINT-01 is a syntax-preserving conversion, not an API-pruning pass).
  - `replaceElement` is the one method whose body is behaviorally sensitive (PSI-tree mutation with a
    3-step fallback chain); its exact fallback order must be preserved verbatim тАФ see design.md ┬з3.2.
- **Commented-out dead code to delete, not port** тАФ confirmed by direct read of
  `src/main/java/net/internetisalie/lunar/lang/psi/LuaPsiUtils.java`. **Three** commented blocks, NOT
  a single contiguous `283-401` range:
  - **lines 183-198** тАФ `getCoveringPsiElement` (fully commented)
  - **lines 283-349** тАФ `isLValue`, `checkForErrors`, and the duplicate `toPsiElementArray` stubs
    (all fully commented)
  - **lines 377-400** тАФ `LuaBlockReturnVisitor` (fully commented)

  > тЪа **Do NOT delete the range `283-401` wholesale.** Lines **350-375 are LIVE public methods** тАФ
  > `toPsiElementArray` (350-359) and `hasDirectChildErrorElements` (361-375) тАФ that are part of the
  > 14-method public API and **must be preserved** (see design.md ┬з2.2 Key API). Only the three
  > commented blocks above are removed.
- **Wildcard import to remove**: `import com.intellij.psi.*;` (line 21) тАФ replace with explicit
  imports for the `com.intellij.psi.*` symbols actually used in the **live** body: `PsiElement`,
  `ResolveState`, `FileViewProvider`, `PsiErrorElement`. (`PsiReference`/`Condition`/`PsiTreeUtil`
  appear only in the deleted comment blocks тАФ drop them; do not carry forward.)
- **Target**: `src/main/kotlin/net/internetisalie/lunar/lang/psi/LuaPsiUtils.kt`,
  `object LuaPsiUtils { @JvmStatic fun nodeType(element: PsiElement): IElementType? = element.node?.elementType }`
  (Elvis/safe-call form, no `!!`) тАФ see design.md ┬з2.2 for every method's converted signature.

### MAINT-01-06: `LuaPluginDisposable.java` тЖТ `LuaPluginDisposable.kt`
- **Current**: `src/main/java/net/internetisalie/lunar/LuaPluginDisposable.java` тАФ
  `@Service({Service.Level.APP, Service.Level.PROJECT}) public final class LuaPluginDisposable
  implements Disposable`, two static factory overloads тАФ `getInstance(): Disposable` (delegates to
  `ApplicationManager.getApplication().getService(LuaPluginDisposable.class)`) and
  `getInstance(Project): Disposable` (delegates to `project.getService(LuaPluginDisposable.class)`) тАФ
  and an empty `dispose()`.
- **Confirmed no callers**: `grep -rn "LuaPluginDisposable"` across `src/` returns only the file's
  own definition. No `plugin.xml` `<applicationService>`/`<projectService>` entry references it
  (`@Service` is itself the declarative registration; none must be added).
- **Must remain a real `class`, not an `object`**: the platform instantiates one app-level and one
  per-project instance via reflection (requires a public no-arg constructor) тАФ a Kotlin `object`
  (single JVM-wide singleton) cannot represent "one instance per project."
- **Target**: `src/main/kotlin/net/internetisalie/lunar/LuaPluginDisposable.kt`,
  `@Service(Service.Level.APP, Service.Level.PROJECT) class LuaPluginDisposable : Disposable { ... }`
  with both `getInstance()` overloads as `@JvmStatic` `companion object` functions тАФ see design.md
  ┬з2.6 for the exact signatures.

### MAINT-01-01 / MAINT-01-05 (DEFERRED тЖТ MAINT-19)
Not specified here. The token-constant interfaces `LuaTokenTypes` / `LuaCatsTokenTypes` are consumed
by the generated lexers via `implements`-inheritance of bare constants (see
[Out of Scope](#out-of-scope)); they cannot be ported to idiomatic Kotlin without flex edits +
manual lexer regeneration, which the **MAINT-19 platform.syntax migration** owns.

## Behavior Rules
- **No call-site changes required.** Each of the four conversions must compile against existing
  consumers тАФ in particular the generated `LuaElementTypes.java` / `LuaCatsElementTypes.java`
  (`new LuaTokenType(...)` / `new LuaCatsElementType(...)`) тАФ with **zero edits to those files**. If
  a conversion would require a consumer edit, the conversion shape (object vs class) is wrong тАФ
  resolve it in design.md, not by patching consumers. (This is exactly why `LuaTokenTypes`/
  `LuaCatsTokenTypes` are deferred: their only conversion shapes *would* force consumer/`.flex`
  edits.)
- **No change to element-type construction cardinality.** Every `IElementType` subclass instance
  (`LuaTokenType`, `LuaCatsElementType`) must continue to be constructed exactly once per distinct
  token, at static/class-load time тАФ never inside a method invoked per-parse or per-file. Direct
  mitigation for the documented `ArrayIndexOutOfBoundsException`-on-registry-exhaustion lesson.
- **Ordering of conversion** (see implementation-plan.md): convert the leaf `IElementType`
  subclasses (`LuaTokenType`, `LuaCatsElementType`) first, then `LuaPsiUtils`. `LuaPluginDisposable`
  has no dependency on any other file in this feature and may be converted in any order.

## Non-Functional Requirements
- **Threading**: none of the four files perform PSI mutation off a write action or I/O on the EDT;
  the conversion must not introduce any (`LuaPsiUtils.replaceElement` already assumes a
  caller-owned `WriteCommandAction` тАФ preserve that contract, do not add an internal wrapper; see
  design.md ┬з2.2).
- **Element-type registry size**: see Behavior Rules тАФ construction cardinality must not change.
- **Memory**: none of these files retain `Project`/`Editor`/`PsiFile`/`VirtualFile` references
  today; the conversion must not introduce any.

## Dependencies
- None of the four files depend on any other in-flight feature; this is a self-contained,
  zero-functional-change conversion.
- **Sibling/deferral edge:** MAINT-01-01 and MAINT-01-05 are carved out to **MAINT-19**
  (platform.syntax migration). MAINT-19 depends on nothing in this feature.
