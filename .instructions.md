# AI Agent Instructions for Lunar

## Project Overview

**Lunar** is a Lua language plugin for the IntelliJ Platform (IDEs like IntelliJ IDEA, PyCharm, CLion, etc.). It provides comprehensive Lua support including syntax highlighting, code completion, debugging, and static analysis.

### Key Technologies
- **Language**: Kotlin (with some Java)
- **Build System**: Gradle with IntelliJ Platform Gradle Plugin
- **Target**: IntelliJ Platform 2024.1+
- **JVM Target**: Java 21

## Coding Standards

### Kotlin Style
- Use Kotlin idioms; avoid Java-style code in Kotlin files
- Prefer val over var
- Use data classes for simple value objects
- Use extension functions for utility methods
- No unnecessary comments; code should be self-documenting

### Import Management
- Remove unused imports regularly (use IDE inspection or `RemoveUnusedImports` task)
- Organize imports alphabetically within groups
- Avoid wildcard imports except for test DSLs

### Naming Conventions
- Classes: PascalCase (e.g., `LuaLexer`, `LuaStatement`)
- Functions/methods: camelCase (e.g., `tokenizeFile`, `getBindings`)
- Constants: UPPER_SNAKE_CASE
- PSI element prefixes: Use `Lua` prefix (e.g., `LuaStatement`, `LuaExpr`)

## IntelliJ Plugin Development Guidelines

### PSI (Program Structure Interface)
- PSI elements represent the structure of Lua code
- Located in `lang/psi/` package
- Must implement `PsiElement` or appropriate interfaces
- Document with `@see` references to PSI parent/child relationships

### Lexer & Parser
- Lexer: `LuaLexer` tokenizes input into `LuaElementTypes`
- Parser: Located in `lang/parser/`, generates PSI tree
- Token types: Defined in `LuaElementTypes.kt` and `LuaTokenTypes.kt`

### Indexing & Resolution
- Symbol resolution uses PSI stubs and caching
- `StubIndex` for fast symbol lookups
- `CachedValuesManager` for caching bindings and references
- Document scopes: Early-bound (locals) vs Late-bound (globals)

### Structure View
- `StructureViewBuilder` provides outline view in IDE
- Implement `StructureViewTreeElement` for PSI elements
- Located in `lang/structure/`

### Debugging
- Debug adapter in `run/` package
- Supports Lua 5.1+ with DBGp protocol
- Remote debugging over TCP

### Settings
- Application settings: IntelliJ Platform SDK, interpreter detection
- Project settings: Language level (5.1-5.4), Runtime configuration
- Settings panels: Located in `settings/` package

## Documentation Languages Supported

### LuaDoc
- Traditional Lua documentation format (IDLua-style)
- Provides inline documentation
- Located in `luadoc/` package

### LuaCATS
- Modern Lua type annotation system
- Supports type hints, overloads, and generics
- Located in `luacats/` package

## Common Tasks

### Adding a New Inspection/Quick Fix
1. Create inspection class extending `LocalInspectionTool`
2. Implement `buildVisitor()` to traverse PSI
3. Register in `plugin.xml` under `<inspectionTool>`
4. Create corresponding quick fix class if applicable

### Adding a New Language Feature
1. Update lexer in `lang/lexer/LuaLexer.flex` if needed
2. Add token types to `LuaElementTypes` or `LuaTokenTypes`
3. Update parser grammar
4. Create PSI element classes in `lang/psi/`
5. Add syntax highlighting in `LuaHighlight.kt`
6. Update documentation if user-facing

### Debugging the Plugin

#### Basic Setup
1. Run Gradle task: `./gradlew runIde`
2. Opens test IDE sandbox with plugin loaded
3. Use IDE debugger to step through code
4. Check `idea.log` in sandbox for errors

#### Advanced Debugging Workflow
When diagnosing complex issues (especially debugger protocol problems):

1. **Add diagnostic logging** to key methods:
   - Use `log.warn()` for user-visible issues (info level logs don't appear in IDE)
   - Include contextual data: input values, state transitions, response content
   - Example: `log.warn("execute() response: ${text.take(200)}")`

2. **Build and run the IDE**:
   ```bash
   ./gradlew build runIde
   ```

3. **Reproduce the issue in the sandbox IDE**:
   - Create a minimal test case that triggers the problem
   - Execute the code path that needs debugging
   - Note any error messages or unexpected behavior

4. **Check the logs**:
   - Sandbox logs: `build/idea-sandbox/GO-*/log/idea.log`
   - Review your diagnostic log statements
   - Look for stack traces and error patterns

5. **Analyze and iterate**:
   - Based on log output, refine your understanding of the issue
   - Add more specific logging if needed
   - Make targeted code changes
   - Rebuild and retest

6. **Clean up logging**:
   - Remove temporary diagnostic logs once root cause is found
   - Keep only essential logging for production
   - Use appropriate log levels (warn for important issues, error for exceptions)

#### Example: Debugging Expression Evaluation
This workflow was used to fix expression evaluation returning wrong types:
- Added logging to see raw debugger responses: `log.warn("execute() raw response: ...")`
- Discovered responses were stringified: `do local _={"1","3"};return _;end`
- Realized we needed to parse stringified values as Lua to recover types
- Implemented `parseStringAsLuaValue()` to parse string content as Lua expressions
- Removed diagnostic logs after fix was verified

### Running Tests

   This should be used when not verifying all tests.

## Performance Considerations

- Use **stubs** for heavy parsing (symbol indexing)
- Use **CachedValuesManager** for expensive computations on PSI nodes
- Avoid blocking the EDT (Event Dispatch Thread)
- Use **background tasks** for long operations
- Profile with IDE's built-in profiler before optimizing

## Common Issues & Solutions

| Issue | Solution |
|-------|----------|
| "Plugin not loading" | Check `plugin.xml` syntax; verify IDE version compatibility in `gradle.properties` |
| "PSI tree broken" | Ensure lexer/parser align; check `LuaElementTypes` definitions |
| "Slow indexing" | Add stub filtering; check `CachedValuesManager` invalidation |
| "References not resolving" | Verify `PsiReference` implementation; check scope binding logic |
